# 15 Apply Wizard 資源申請

## 1. 功能目的

Apply Wizard 用於讓使用者提出 HCM 資源配置申請。申請人可以選擇既有專案或建立申請用的新專案資料，指定專案下需要納管的系統，並為每個系統選擇 Dedicated Pool 或 Shared Pool 資源。

本功能的重點不是直接建立雲端資源，而是產生一筆可追蹤、可審核、可執行的配置申請。申請送出後，後續由 Allocation Management 或相關作業流程接續處理。

若申請中填寫新專案或新系統，Apply Wizard 只將其作為申請內容保存；正式 Project/System 主檔在 Allocation Configure 或 Execute 生效時建立或確認。

Apply Wizard 不直接呼叫 Provider 外部 API。畫面上可選擇的雲端、資源池、容量資訊，來自 Cloud Settings 與 Provider Plugin 已同步回 HCM 的資料。

## 2. 使用者與權限

| 角色 | 可執行動作 | 說明 |
| --- | --- | --- |
| 一般使用者 | 建立申請、查詢自己的申請紀錄 | 只能看到自己可使用的專案與自己送出的申請 |
| 專案管理者 | 建立申請、查詢相關專案申請紀錄 | 可依組織規則查看或建立自己負責專案的申請 |
| HCM 管理者 | 建立申請、查詢所有申請紀錄 | 可協助使用者建立申請，也可檢視全部申請狀態 |

## 4. 主要業務資料異動情境

### 4.1 建立資源配置申請

使用者完成 Project、Systems、Pools、Confirm 後送出申請。HCM 建立一筆配置申請，內容包含申請人、專案、系統、Dedicated Pool 申請、Shared Pool 申請與資源配額。

送出後，申請狀態預設為待處理，並出現在申請紀錄中。後續審核、配置、完成或退回，由 Allocation Management 或其他後續流程處理。

### 4.2 新專案與新系統資料

若使用者在申請流程中輸入新專案或新系統，這些資料會成為該筆申請的一部分。是否在送出時立即建立正式 Project/System 主檔，或等待申請核准後才建立，需由 HCM 的業務流程決定。

在本規格中，Apply Wizard 僅定義使用者可以在申請中帶入新專案與新系統資料，不定義其主檔生效時點。

### 4.3 Dedicated Pool 申請

Dedicated Pool 申請代表使用者希望將指定的專用資源池配置給某個系統。此類申請通常不需要在申請畫面填寫 CPU、Memory、Disk 配額，因為資源池本身即代表可配置範圍。

### 4.4 Shared Pool 申請

Shared Pool 申請代表使用者希望從共享資源池中取得部分資源。使用者需填寫 CPU、Memory、Disk 等配額，HCM 依據目前已配置、待處理申請與本次申請草稿，顯示剩餘可申請容量。

## 5. Provider Plugin 掛點

Apply Wizard 不直接掛接 Provider Plugin，也不直接呼叫 Provider 外部 API。它使用的 Provider 相關資訊，來自其他功能事先同步並整理到 HCM 的業務資料。

| 使用位置 | Provider 影響 | 資料來源 | 本文件描述方式 |
| --- | --- | --- | --- |
| 可用雲端 | 不同 Provider 可能有不同雲端連線與啟用狀態 | Cloud Settings | 僅說明使用已啟用雲端 |
| 可用 Pool | 不同 Provider 可同步出的資源池屬性可能不同 | Pool 資料 | 僅說明 Dedicated/Shared 可選規則 |
| 容量資訊 | 不同 Provider 的容量單位或可用量定義可能不同 | Provider Plugin 同步後的 HCM Pool 資料 | 以 HCM 標準欄位顯示 |
| 額外申請限制 | 不同 Provider 可能有額外的資源選擇限制 | Provider Plugin 差異規格 | 在 Provider 差異文件描述 |

若某個 Provider 對申請資源有特殊規則，例如特定 Pool 需要指定 Region、Storage Policy、Network Profile 或其他條件，應在該 Provider Plugin 規格中描述如何影響可選資料與提示文字。Apply Wizard 本文只保留共通流程。

| Pool 顯示名稱 | 畫面顯示 HCM Pool 名稱與雲端資訊 | 是否需顯示 Provider 特有識別資訊 |
| 容量欄位 | HCM 顯示 CPU、Memory、Disk | Provider 原始容量欄位與單位轉換 |
| 可申請條件 | 僅顯示啟用雲端與可用狀態的 Pool | Provider 是否有額外的可申請條件 |
| Shared Pool 配額 | 使用 CPU、Memory、Disk 填寫申請量 | Provider 是否需補充其他配額維度 |
| Dedicated Pool 配置 | 申請整個指定 Pool | Provider 是否需補充整池配置的業務限制 |

## 7. 下游資料流

| 送出資料 | 下游功能 | 用途 |
| --- | --- | --- |
| 配置申請 | Allocation Management | 審核、配置與更新申請狀態 |
| 申請歷程 | Apply History | 讓申請人查詢申請狀態 |
| 專案資料 | Project Dimension | 若申請包含新專案，後續可能形成專案主檔 |
| 系統資料 | Project/System 管理流程 | 若申請包含新系統，後續可能形成系統主檔 |
| Pool 配置需求 | Resource Overview / Allocation Management | 呈現待處理與已配置資源狀態 |

## 8. 待確認事項

| 項目 | 說明 | 建議確認方向 |
| --- | --- | --- |
| 新專案生效時點 | Apply Wizard 可填寫新專案，但是否送出即建立正式主檔尚需確認 | 建議由審核流程決定，避免未核准資料污染主檔 |
| 新系統生效時點 | Apply Wizard 可填寫新系統，但是否送出即建立正式主檔尚需確認 | 建議與新專案採同一套生效規則 |
| Dedicated Pool 是否需估算容量 | 目前申請畫面以整池申請為主，不要求填寫配額 | 若業務需要評估成本，可補充估算欄位 |
| History 狀態分類 | 畫面有 Pending / Done 分類，但完整狀態可能更多 | 建議由 Allocation Management 統一定義狀態 |
| Provider 額外限制 | 目前 Apply Wizard 使用 HCM 標準 Pool 資料 | 若 Provider 有額外限制，放入 Provider Plugin 差異文件 |
